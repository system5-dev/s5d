# unit-tests

Stack-agnostic unit-test installer + runner + scaffolder. Same shape as
`code-quality`: detect → install → write configs with **mandatory coverage
thresholds and reports** → run → report.

Sixteen stacks recognised — `js/ts, python, go, rust, java, kotlin, dotnet,
ruby, php, swift, shell`. SoTA framework + coverage tool per stack
documented in `references/catalog.md` (last verified 2026-05-28).

## What's mandatory

- **Coverage threshold** — every stack ships with a default 70 % line-coverage
  gate baked into its native config (vitest thresholds, pytest `--cov-fail-under`,
  go tool cover %, cargo-llvm-cov `--fail-under-lines`, JaCoCo `<rule>`,
  Kover `minBound`, coverlet `<Threshold>`, SimpleCov `minimum_coverage`,
  PHPUnit coverage-check). Override with `--threshold N`.
- **Three report formats** — LCOV (for Codecov / Coveralls / VS Code), JUnit
  XML (for CI annotations), HTML (for local dev). Plus console summary.
- **All reports under `test-reports/<stack>/`** — single root so CI artefact
  uploads, .gitignore, and tooling can target one path.

## Flow

```
1. scripts/detect.sh           → JSON: stacks + framework + coverage tool + test counts
2. scripts/setup.sh --auto     → install configs from templates/<stack>/ with threshold substitution
3. scripts/write.sh <src>      → scaffold a test file for one source file
4. scripts/run.sh              → run + emit LCOV / JUnit / HTML + enforce coverage gate
5. CI workflow generated by setup.sh --ci github (or gitlab)
```

## SoTA framework matrix (May 2026)

| Stack | Test framework | Coverage tool |
|---|---|---|
| JS/TS | **Vitest 4** (Jest 30 for React Native / legacy) | v8 (built into Vitest) |
| Python | pytest | pytest-cov + coverage.py |
| Go | built-in `go test` | built-in `go test -cover` + gcov2lcov for export |
| Rust | built-in `cargo test` | **cargo-llvm-cov** (tarpaulin only for legacy ptrace) |
| Java | JUnit 5 | JaCoCo |
| Kotlin | JUnit 5 or Kotest | **Kover** (JetBrains, replaces JaCoCo for Kotlin DSL) |
| .NET | xUnit | coverlet (built into `dotnet test`) |
| Ruby | RSpec | SimpleCov |
| PHP | PHPUnit | Xdebug or PCOV (driver) + coverage-check |
| Swift | **Swift Testing** (new in Swift 6) + XCTest for UI | `xcrun llvm-cov` |
| Shell | bats-core | kcov (optional) |

Why these choices: see `references/catalog.md`. The catalog also tracks
when to keep a legacy framework (Jest if you're on React Native, XCTest
if you have UI/perf tests, JaCoCo if your Kotlin project is mostly Java).

## Determinism boundary

| Step | In script | In agent |
|---|---|---|
| Stack + framework detection | ✓ `detect.sh` | — |
| Coverage tool detection | ✓ `detect.sh` | — |
| Config file generation with threshold substitution | ✓ `setup.sh` | — |
| CI workflow emission | ✓ `setup.sh --ci` | — |
| Choosing migration path (Jest→Vitest, JaCoCo→Kover, XCTest→Swift Testing) | — | ✓ |
| Resolving existing-config collisions | — | ✓ |
| Test file scaffolding (stub copy + substitution) | ✓ `write.sh` | — |
| Filling TODO scenarios in the stub with real assertions | — | ✓ |
| Running tests + producing reports | ✓ `run.sh` | — |
| Interpreting coverage gap output to suggest where to add tests | — | ✓ |

## Hard rules

- **Coverage and reports are non-negotiable.** Every stack gets all three
  formats (LCOV + JUnit + HTML) plus a console summary. A framework setup
  without coverage is rejected — bump the threshold to 0 if you need to
  ship without enforcement, but the reports still run.
- **Never silently replace an existing test framework.** Detect what's
  there; present migration path; wait for confirmation.
- **Never invent passing tests.** `write.sh` produces stubs with `TODO`,
  `it.todo`, `markTestIncomplete`, or `@should_panic("TODO")` so the test
  fails until the agent / human fills it in. Stubs that pass-by-default
  hide real coverage gaps.
- **Test file naming follows stack convention.** `write.sh` picks the
  canonical path: `<src>.test.ts` next to source for JS, `test_<src>.py`
  for Python, `<src>_test.go` for Go, `src/test/java/...` mirror for Java,
  etc. Don't fight the framework.
- **Coverage thresholds in the native config, not in a wrapper.** The
  config is where the team will look first; surface the gate there so
  CI failure modes are obvious.
- **Reports under `test-reports/<stack>/`** — add this to `.gitignore`.
  The skill writes nothing there until `run.sh` produces output.
- **Honour existing CI.** Generate to `.github/workflows/unit-tests.yml`
  (separate from any code-quality.yml or existing ci.yml). User merges
  later if they want one workflow.

## Worked example

```bash
# 1. What's here?
bash ~/.agents/skills/unit-tests/scripts/detect.sh

# 2. Install configs + CI workflow for everything detected (default 70% threshold)
bash ~/.agents/skills/unit-tests/scripts/setup.sh --auto --ci github

# 3. Scaffold a test file for a source file you don't yet have tests for
bash ~/.agents/skills/unit-tests/scripts/write.sh lib/server-pricing.ts
# → writes lib/server-pricing.test.ts with TODO markers

# 4. Fill in the TODOs (human or LLM)

# 5. Run + emit reports
bash ~/.agents/skills/unit-tests/scripts/run.sh
# → test-reports/js/{coverage.lcov, junit.xml, coverage/index.html}
# → exits non-zero if coverage < threshold

# 6. View HTML locally
open test-reports/js/coverage/index.html
```

For a stricter run:
```bash
bash ~/.agents/skills/unit-tests/scripts/setup.sh --auto --threshold 85 --force
bash ~/.agents/skills/unit-tests/scripts/run.sh --threshold 85
```

## When NOT to use

- The project has a mature, well-maintained test suite with reports already.
  Targeted fixes (raise threshold, switch reporter) are cheaper than a
  full skill run.
- You need integration / e2e tests against real services. This skill is
  unit tests only; e2e is Playwright / Cypress / TestCafé territory.
- The framework requires a database, mock server, or fixture setup that
  the skill can't infer from the repo. Run the skill to install the
  framework, then wire fixtures manually.
