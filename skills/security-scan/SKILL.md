# security-scan

Category-agnostic security-scanner installer + runner + triage. Same shape as
`unit-tests` and `code-quality`: detect → install configs → run → gate on
severity → emit **SARIF + SBOM + JUnit** → dedup CVEs → manage waivers.

Seven categories recognised — `sast, sca, secrets, container, iac, license,
sbom`. SoTA tool per category documented in `references/catalog.md`
(last verified 2026-05-28). The skill writes **tool-native configs** and prints
install one-liners; it does **not** install scanner binaries.

## What's mandatory

- **Severity gate per category** — every run computes a severity for each
  finding and fails the build when any finding meets `--fail-on` (default
  `high`). SAST gates on ERROR/high+, SCA on CVSS ≥ 7, **secrets are
  zero-tolerance** (any verified secret fails). Drop the gate only with
  `--no-gate` — the reports still run.
- **SARIF 2.1.0 is the primary report** — one `findings.sarif` per category
  plus a merged `all.sarif`. GitHub renders SARIF natively in the Security tab;
  CI uploads it via `codeql-action/upload-sarif`.
- **SBOM in CycloneDX** — `sbom.cdx.json` emitted by Syft from the same
  lockfiles SCA scans. Sign with Cosign in CI when keys are present.
- **Waivers are explicit, never silent** — a suppressed finding requires a
  `.s5d/security-waivers.yaml` entry with rule-id, reason, condition, approver,
  and expiry date. No `# nosec`-style inline silencing. **S5D canonical waiver
  (per decision.skill-suite-integration):** when an S5D spec gates the work, the
  authoritative waiver is the MCP `s5d_waiver` tool (gate waivers are MCP-only —
  no CLI verb). This skill's `.s5d/security-waivers.yaml` is the scanner-local
  fallback for repos not running through S5D's MCP.
- **All reports under `test-reports/security/<category>/`** — single root so CI
  artefact uploads, `.gitignore`, and triage can target one path.

## Flow

```
1. scripts/detect.sh            → JSON: which of 7 categories apply + tool + evidence
2. scripts/setup.sh --auto      → write native configs from templates/<category>/ + waivers store
3. scripts/run.sh               → run scanners + emit SARIF/SBOM/JUnit + enforce severity gate
4. scripts/triage.sh            → dedup CVEs across scanners (text | md | json)
5. scripts/waiver.sh add|list|check → manage suppressions with rationale + expiry
6. CI workflow generated by setup.sh --ci github (or gitlab)
```

## SoTA category matrix (May 2026)

| Category | Tool | Fallback | Gate |
|---|---|---|---|
| SAST | **Semgrep** (`--config auto`) | CodeQL | ERROR / high+ |
| SCA (deps) | **Trivy fs** `--scanners vuln` | osv-scanner, Dependabot | CVSS ≥ 7 |
| Secrets | **gitleaks** (history + tree) | trufflehog | zero-tolerance |
| Container | **Trivy image** | Grype | CVSS ≥ 7 |
| IaC | **Trivy config** | Checkov | high+ |
| License | **Trivy** `--scanners license` | scancode | policy-defined |
| SBOM | **Syft** → CycloneDX (+ Cosign) | cdxgen | n/a (artefact) |

Why these choices: see `references/catalog.md`. Trivy is the spine — one binary
covers SCA, container, IaC, and license, so a JS/TS repo needs only Semgrep +
Trivy + gitleaks + Syft. DAST (OWASP ZAP + Nuclei) and PII (Microsoft Presidio)
are catalogued as v1.1 follow-ups, not wired into the default run.

## Hard rules

- **The severity gate is non-negotiable.** Every category computes a severity
  and the run fails on `--fail-on` or higher. Secrets are zero-tolerance. Use
  `--no-gate` to report-without-failing, never to skip the scan.
- **Never silently suppress a finding.** Suppression goes through
  `waiver.sh add` → `.s5d/security-waivers.yaml` with reason, condition,
  approver, and a future expiry date. Expired waivers stop suppressing
  automatically; `run.sh` re-surfaces the finding.
- **SARIF is the contract.** Every scanner output normalises to SARIF 2.1.0 so
  GitHub, triage, and waivers all read one schema. A scanner without a SARIF
  formatter is wrapped, not exempted.
- **Never apply configs to a repo you're only auditing.** `setup.sh` writes
  into the target repo; smoke-test into a throwaway dir first. The skill does
  not run scanners on code it was only asked to read.
- **Don't install binaries silently.** `setup.sh` writes configs and prints
  `install.md` one-liners (brew / curl). The human or CI installs the tool.
- **Reports under `test-reports/security/`** — add to `.gitignore`. The skill
  writes nothing there until `run.sh` produces output.
- **Honour existing CI.** Generate to `.github/workflows/security-scan.yml`
  (separate from any ci.yml / unit-tests.yml). User merges later if they want
  one workflow.

## Worked example

```bash
# 1. What categories apply here?
bash ~/.agents/skills/security-scan/scripts/detect.sh

# 2. Write configs + waivers store + CI workflow for everything detected
bash ~/.agents/skills/security-scan/scripts/setup.sh --auto --ci github

# 3. Run scanners, emit reports, enforce the gate (fails on high+ by default)
bash ~/.agents/skills/security-scan/scripts/run.sh
# → test-reports/security/{sast,sca,secrets,...}/findings.sarif
# → test-reports/security/all.sarif, sbom.cdx.json, junit.xml
# → exits non-zero if any finding ≥ high

# 4. Dedup CVEs across SCA + container into one PR-comment table
bash ~/.agents/skills/security-scan/scripts/triage.sh --format md

# 5. A finding is a known false positive — waive it with rationale + expiry
bash ~/.agents/skills/security-scan/scripts/waiver.sh add CVE-2025-12345 \
  --reason "transitive dep, not reachable from our code path" \
  --condition "until upstream patch lands" \
  --approved-by rvoronin --expires 2026-08-31
bash ~/.agents/skills/security-scan/scripts/waiver.sh list --active
```

For a stricter run (fail on any medium, scan only two categories):
```bash
bash ~/.agents/skills/security-scan/scripts/run.sh --fail-on medium --categories sast,secrets
```

## When NOT to use

- The repo already has a mature scanning setup (CodeQL + Dependabot + a secret
  scanner) wired into CI with triage. Targeted additions are cheaper than a
  full skill run.
- You need **DAST** (running-app scanning) or **PII discovery** in data stores.
  v1 wires only static categories; DAST (ZAP + Nuclei) and PII (Presidio) are
  catalogued as follow-ups — see `references/catalog.md`.
- You need a one-off scan of a single file. Run Semgrep or Trivy directly; the
  skill's value is the config + gate + waiver + CI lifecycle, not the binary.
