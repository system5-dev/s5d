S5D

A control plane for repository changes with AI participation.

When AI agents write code in your repo, three questions get hard to answer: what was the system supposed to become, why was this approach chosen over the alternatives, and does the code still match that decision? S5D keeps the answers in your repository, next to the code, and verifies them mechanically.

It is not a replacement for git, tests, or planning tools. It adds four things on top of normal development:

1. Target state

Describe what the system should become and which existing architecture it reuses — before writing code.

2. Explicit choice

Compare genuinely different alternatives before committing to one. Decisions are recorded, not implied.

3. Run evidence

Record what agents and tools actually produced — with an approval chain and SHA256 integrity binding.

4. Verify in code

Check that the code still matches the decision. Trace, reconcile, or roll back when it doesn't.

Alpha. S5D is experimental. The core state machine is tested, but command surface and file formats may still change between releases.

Install

git clone https://github.com/system5-dev/s5d.git
cd s5d
./install.sh

install.sh must be run from a checked-out copy of the repository. curl | bash is intentionally unsupported.

Initialize your repo

cd your-project
s5d init

This creates the .s5d/ directory. If the repo has .git/, init also installs a pre-commit hook that keeps specs and code in sync (manual entrypoint: s5d hook pre-commit).

Connect your AI agent

S5D runs as an MCP server, so coding agents can drive it through tools instead of shelling out:

# Claude Code only
s5d init --claude

# All supported agents (Claude Code, Cursor, Codex, Gemini CLI)
s5d init --all

S5D is also distributed as a plugin for Claude Code, Gemini CLI, and Codex. init additionally writes a short S5D usage block into your repo's AGENTS.md so any agent that reads it knows the rules.

Stay up to date

s5d admin update check
s5d admin update apply

Quick start

The minimal loop — one spec, one approval, one import:

# Create a spec scaffold
s5d new feat.my-feature --product myapp

# Edit the spec YAML, then validate and preview
s5d verify validate .s5d/packages/feat.my-feature__*.s5d.yaml
s5d state preview  .s5d/packages/feat.my-feature__*.s5d.yaml

# A human approves the plan
s5d state approve  .s5d/packages/feat.my-feature__*.s5d.yaml --reviewer yourname

# Implement the code, run the gates, import the result
s5d verify run-gates .s5d/packages/feat.my-feature__*.s5d.yaml
s5d state import     .s5d/packages/feat.my-feature__*.s5d.yaml --verified-by verifiername

# Later: check nothing drifted
s5d state drift-check
Two names, on purpose. The person who approves (--reviewer) and the one who verifies the import (--verified-by) must differ. A common split: the human approves, the agent verifies.

Closing the loop after shipping is optional but recommended:

s5d state reflect .s5d/packages/feat.my-feature__*.s5d.yaml \
  --summary "Telemetry stayed inside target bounds" \
  --verdict confirmed \
  --measurement-window "7d post-ship" \
  --telemetry "grafana://my-dashboard" \
  --heuristic "Keep rollout verdicts tied to explicit telemetry refs"

Core concepts

Spec and record

Every governed change produces two files: a spec (what you intend — the target state, acceptance criteria, chosen approach) and a record (what actually happened — approvals, gate results, run evidence). Both live in .s5d/ and travel with the repo.

The SHA256 chain

The CLI binds spec and record cryptographically:

spec → preview → approve → import → drift-check
                                         ↓
                                    reconcile / rollback

Approval signs the exact spec content. If anyone edits the spec after approval, the hash no longer matches and import refuses — you re-preview and re-approve. drift-check later compares the live state against the last applied fingerprint; reconcile restores it, rollback reverses an import.

Human-in-the-loop, machine-held state

Two steps can never be waived or automated away: a human confirms decisions, and a human approves runs. Everything around them — validation, graph checks, hashing, drift detection — is mechanical. Human-facing summaries, phase names, and ETAs are derived explanations; the machine-readable files are the source of truth.

The flow

Route→ Discover→ Target→ Decide→ Spec→ Run→ Verify→ Ship→ Learn
StageWhat happens
RouteClassify the request: which tier, which mode, where to enter the flow. Small fixes exit here — they don't need S5D.
DiscoverMap the project once: sources, domains, capabilities, components. Skipped on follow-up work when the map is current.
TargetState what should change and define acceptance criteria before looking at options.
DecideCompare at least three genuinely different approaches, challenge the favorite, record the winner. A human confirms.
SpecTurn the decision into a feature spec: problem, acceptance scenarios, implementation approach, DO/DON'T boundaries.
RunPreview, human approval, then implementation — by you or by a bounded agent run with captured output.
VerifyGates run (schema, graph, tests, lint — whatever the spec declares), then import seals the SHA256 chain.
ShipPush and deploy — each with explicit human permission.
LearnReflect with real telemetry: confirmed, refuted, or iterate. Reusable heuristics get recorded.

If you arrive with a decision already made (“implement X, we agreed on the approach”), the flow enters directly at Spec — the skipped stages are recorded as an explicit waiver pointing at the prior decision.

Tiers & gates

Not every change deserves the same ceremony. S5D classifies work into tiers:

TierWhenDefault gates
NoteCapture context, one-shot thoughtNone
DecisionCompare alternatives, record an architectural choiceNone
LightweightSimple feature, single domainSchema
StandardRegular feature, two or more domainsSchema + Graph
HighAuth, payments, PII, complianceSchema + Graph, no waivers

Gates are checks a spec must pass before import. Schema, graph, and architecture gates are built in; add lint, test, or contract gates by configuring their commands in .s5d/config.yaml. On High tier, assurance gates can never be waived.

Out of scope entirely: bug fixes under 30 lines, config-only and docs-only changes. Just make them — S5D deliberately stays out of the way.

Worked example

A full decision-to-feature pass: choosing a refresh-token rotation strategy, then implementing it.

s5d init
s5d new decision.refresh-rotation --tier decision --product auth \
  --question "How should refresh tokens rotate?"

# Add competing approaches (at least 3, different in kind)
s5d decision add-hypothesis <spec> --title "Server-side rotation" \
  --content "Rotate on every refresh, persist token family state" --scope "auth boundary"

# Attach evidence per hypothesis
s5d decision add-evidence <spec> --hypothesis-id server-side-rotation \
  --evidence-type internal --content "Revocation lookup <5ms at p95" \
  --verdict pass --formality 4 --claim-scope latency --reliability 0.8

# Create the feature spec linked to the winning hypothesis
s5d new feat.refresh-rotation --tier standard --product auth \
  --hypothesis-id server-side-rotation
s5d verify validate <feature-spec>

# Record the decision (human confirms)
s5d decision decide <decision-spec> --title "Use server-side rotation" \
  --winner server-side-rotation --confirmed-by Roman \
  --rationale "Best revocation/complexity balance" \
  --consequences "Needs persistent token-family store"

# Approve, implement, verify, import
s5d state preview <feature-spec>
s5d state approve <feature-spec> --reviewer Roman
# ... implement the code ...
s5d verify run-gates <feature-spec>
s5d state import <feature-spec> --verified-by Diana

# Close the loop
s5d state reflect <feature-spec> --summary "Shipped cleanly" \
  --heuristic "Link winner to feature spec before decide"

Agent run control

S5D records agent execution as evidence against a desired state — an agent finishing is never the same as the work being accepted.

# See executable work states for a spec, start one
s5d run list  <spec>
s5d run start <spec> --id prototype

# Emit a bounded task package for an engine
s5d run task <spec> --phase prototype --engine ralph

# Execute an approved engine; stdout/stderr captured, output hash recorded
s5d run exec <spec> --id prototype --engine local-engine

# A human explicitly accepts the result
s5d run accept <spec> --id prototype --reviewer yourname
  • run exec only runs engines declared and approved in .s5d/config.yaml; output lands under .s5d/runs/ with its hash recorded.
  • run task --engine ralph emits a bounded task package (modes: init for context warm-up, bugfix for regression-first fixes); packages persist under .s5d/tasks/.
  • Harness adds an operational layer for longer runs: s5d run harness start / status / exec gives the engine an isolated git worktree, a clean-state preflight, heartbeat, timeouts, and a journal under .s5d/harness/.
  • Harness state is never run truth — the record file stays authoritative for evidence, gates, and approvals.

Discovery

Before the first governed change in a repo, S5D maps it. Discovery produces two human-authored artifacts in .s5d/discovery/ — a source survey (which sources of truth exist and what each is canonical for) and an architecture map (domains, capabilities, entities, components, UX surfaces, cross-domain edges, and open unknowns). Every claim in the map is tagged [VERIFIED], [INFERRED], or [SPECULATIVE].

Alongside the maps, a deterministic index keeps file-level evidence current:

# Build / refresh the technical index (file index, evidence, graph)
s5d discover sync

# Fail if the snapshot is stale (useful in CI)
s5d discover check

The same pair exists for coverage snapshots — s5d codebase sync / check reports how much of the codebase is governed by specs (governed / partial / blind per module).

Editor support

Spec files are YAML with a strict schema. s5d lsp starts a Language Server over stdio that publishes live diagnostics for .s5d.yaml files — parse errors with exact positions, plus semantic validation as you type. Point your editor's LSP client at:

s5d lsp

There is also a pre-commit hook (installed by s5d init in git repos) that validates touched specs before they can be committed.

Gotchas

  • Quote strings containing : in YAML lists. - Backward compat: foo parses as a map, not a string. Quote it or reword with a dash.
  • Required fields cascade. A component requires feature + domain + container + paths; a domain requires product + classification; a capability requires domain. The metamodel reference documents the full cascade.
  • Editing after approval invalidates the approval. Any change to an approved spec changes its hash. Recovery is always: preview → approve → import again.
  • Approver and verifier must differ. state import --verified-by rejects the same name that approved.
  • Decisions need at least 3 hypotheses — and they must differ in kind, not just in parameters. Two-hypothesis decisions are blocked.
  • contract.format is an enum: openapi / json_schema / protobuf / typespec.
  • Review gates on Standard/High tiers close via s5d_waiver (an MCP-only operation), not by attaching evidence.

Non-goals

  • Not a replacement for git
  • Not a replacement for tests
  • Not a planning or project-management system
  • Not required for small fixes — bug fixes under 30 LOC, config-only, docs-only changes just happen
  • Not a code generator
